# 海马体 · 9D 多维情感记忆系统 — 设计文档

> 版本: 1.0 | 最后更新: 2026-05-29

---

## 一、设计理念

### 1.1 核心隐喻

系统以人脑的**海马体**（Hippocampus）为隐喻模型：

| 人脑功能 | 系统对应 |
|:---|:---|
| 感官输入 → 短期记忆 | 用户聊天输入 → extractMemory() |
| 海马体编码 → 长期记忆 | 9D 词元提取 → memory-store.js 持久化 |
| 线索触发 → 回忆重构 | 用户提问 → searchDimension() → AI 用记忆回答 |
| 反复回忆 → 记忆增强 | 同一事件的多次提及 → enrichExisting() 合并强化 |
| 遗忘 | 300 条上限的自动淘汰 |

### 1.2 核心原则

**原则一：原文不可篡改**

> 9D 记忆的 `text` 字段必须存储用户原始输入，AI 不得改写、概括、拼接。
> 这是系统的基石——如果允许 AI 改写记忆内容，就会产生"记忆幻觉"，
> 即 AI 把编造的细节当作事实存储，下次检索时拿出来当真实回答。

**原则二：9D 维度是桥梁，不是终点**

> 9D 拆分不是为了分类存档，而是为了**线索检索**。
> 每条记忆拆成 9 个维度，用户输入任意关键词（场景、人物、情感、物件），
> 系统从 9 个角度匹配，找到最相关的记忆。
> 拆分越精细，多角度匹配的命中率越高。

**原则三：记忆是活的**

> 同一事件被多次提起时，旧记忆会被**增强**（合并新关键词、更新情感强度），
> 而不是覆盖或新建。这样越常回忆的事越鲜明，符合人脑记忆规律。

**原则四：AI 不能猜，只能问**

> 当用户问"还记得那件事吗"且有多条相关记忆时，AI **不得自行选择一条回答**。
> 而是列出每条记忆的辨识线索（场景/人物/物件/情感），反问用户是哪一条。
> 用户确认后，再用该记忆的细节回答。

## 三、9D 数据模型

### 3.1 九维定义

```
┌──────┬─────────────┬────────────────────────────────┐
│ 维度  │   名称      │          作用                   │
├──────┼─────────────┼────────────────────────────────┤
│ X    │ 语义        │ 关键词、主题——"说了什么"        │
│ Y    │ 时间        │ 季节、早晚——"什么时候"           │
│ Z    │ 情感        │ 情绪类型、强度、效价——"什么情绪"  │
│ W    │ 人物        │ 谁在场、什么角色——"和谁"         │
│ V    │ 场景        │ 地点、氛围、光线——"在哪里"       │
│ R    │ 关系        │ 互动类型、亲密度——"什么关系"     │
│ M    │ 深刻度      │ 重要性、保留优先级——"多重要"     │
│ G    │ 物件        │ 物品、象征意义——"有什么"         │
│ S    │ 感官        │ 视/听/嗅/触/味——"感受到什么"    │
└──────┴─────────────┴────────────────────────────────┘
```

### 3.2 为什么是 9 个维度？

这 9 个维度不是随意选的，它们映射了人类记忆编码的几个关键通道：

| 通道 | 维度 | 理论来源 |
|:---|:---|:---|
| 语义记忆 | X | Tulving 1972 — 语义 vs 情景记忆 |
| 情景时间 | Y | 时间轴 — 事件排序的基础 |
| 情绪标记 | Z | Damasio 1994 — 躯体标记假说 |
| 社会关系 | W, R | 人类是社会性动物，记忆围绕人组织 |
| 场景 | V | 情景记忆的核心载体 |
| 意义权重 | M | 越重要的事记得越牢 |
| 物件锚点 | G | 物品是回忆的触发器（Proust效应） |
| 感官沉浸 | S | 感官细节让回忆有"临场感" |

## 四、搜索与排序引擎

### 4.1 设计思路

搜索不是简单的关键词匹配，而是**9D 多维度评分**。每条记忆从 15 个因子获得分数，加权后排序。

```
用户输入: "还记得咖啡厅的事吗？"

评分因子（memory-store.js searchDimension）:
┌──────┬────────────────────────────────┬───────┐
│ 因子  │ 匹配什么                       │  权重  │
├──────┼────────────────────────────────┼───────┤
│  1   │ 标题包含查询词 / 查询包含标题前4字 │ +0.4  │
│  2   │ 文本包含查询词                   │ +0.3  │
│  3   │ 分类匹配                         │ +0.25 │
│  4   │ 关键词双向匹配                   │ +0.2  │
│  5   │ 标签双向匹配                     │ +0.15 │
│  6   │ 场地精准匹配（如"咖啡厅"）       │ +1.0  │
│  7   │ 场地文本匹配                     │ +0.6  │
│  8   │ 季节/时间匹配                    │ +0.25 │
│  9   │ 物件名称匹配                     │ +0.3  │
│ 10   │ 关系类型匹配                     │ +0.25 │
│ 11   │ 人物名称匹配                     │ +0.25 │
│ 12   │ 感官关键词匹配                   │ +0.2  │
│ 13   │ 情感词匹配                       │ +0.2  │
│ 14   │ 情感强度加成                     │ 强度×0.1│
│ 15   │ 记忆深度加成                     │ 深度×0.15│
│ 16   │ 回忆意图加成                     │ 含"记得"等×1.3 │
│ 17   │ 情感查询放大                     │ 含"开心"等再×0.2│
│ 18   │ 时间衰减                         │ 30天内衰减至0.1│
└──────┴────────────────────────────────┴───────┘

→ 总和 > 0.15 的进入候选 → 排序 → Top 5
```

### 4.2 重要设计决策

**为什么场地匹配权重最高（+1.0）？**
- 用户问"咖啡厅那件事"时，场地是最明确的筛选条件
- 相比情感词（"开心"可能匹配很多记忆），场地是唯一性最强的维度
- 这是从实际测试中得出的结论——之前权重 0.5 时咖啡厅记忆排不到第一

**为什么有"回忆意图"加成（×1.3）？**
- 用户说"记得xx吗"时明确在触发回忆，此时应提高匹配灵敏度
- 不加成的话，非回忆类查询（如"咖啡厅的营业时间"）的匹配会被不相关记忆干扰

**为什么有时间衰减？**
- 近期的记忆更可能被用户提及
- 30 天后衰减到 0，但只减 0.1，不影响关键词匹配

## 五、记忆生命周期

### 5.1 创建

```
一次聊天 → extractMemory()
  ├── 用户输入 < 5字 → 跳过
  ├── 用户输入 > 500字 → 跳过（可能是文章全文）
  ├── 以"什么是/怎样/如何/为什么/测试"开头 → 跳过
  ├── LLM 提取 9D 成功 → 用 LLM 结果
  └── LLM 提取失败 → makeBasicMemory() 用关键词生成基础记忆
```

### 5.2 同事件合并

```
新记忆 → enrichExisting()
  ├── 同 venue + 同关键词/人物 + 间隔<7天 → 增强旧记忆，不新增
  │    ├── text: 保留最长的那段原文
  │    ├── keywords: 合并去重
  │    ├── emotion: 加权平均
  │    ├── people: 合并去重
  │    ├── goods: 合并去重
  │    └── importance: 取最大值
  └── 不同事件 → 作为新记忆添加
```

### 5.3 检索

```
用户提问 → search(query, 5)
  ├── searchDimension() → 9D 评分排序
  │   └── 有结果？→ 返回 Top 5
  └── searchRaw() → 纯文本 bigram 匹配（兜底）
      └── 返回至多 3 条
```

### 5.4 消费（AI 回答）

```
5 条记忆 → chatWithDeepSeek()
  1. 注入系统提示词（都灵人格 + 三条行为规则）
  2. 将 5 条记忆格式化为上下文块
  3. 每条记忆标注：
     - 辨识线索（独特人物/物件/关键词）
     - 突出维度（哪几个维度分数最高）
     - 情感强度等级
     - 场景/人物/物件摘要
  4. 附加规则：
     - 多条记忆 → 不猜，列出选项问用户
     - 用户提到地点 → 只从那地点的记忆中选择
     - 无匹配 → 坦诚说不记得，引导用户分享
  5. 最后一条 prompt 强调用户问题 + 地点优先级
  6. 调用 DeepSeek API
```

### 5.5 遗忘（淘汰机制）

```
当前机制：固定 300 条上限
  ├── 聊天记忆（priority: 1）先被淘汰
  └── 文章/知识库（priority: 2）后被淘汰

理想机制（待实现）：
  ├── 按 priority 分层淘汰
  ├── 低情感强度（intensity < 0.3）的优先淘汰
  └── 长时间未提及的（30天+）优先淘汰
```

## 六、规则系统（AI 行为约束）

这是整个系统最关键的创新——通过提示词工程，用**硬规则**约束 AI 的记忆行为。

### 6.1 永久规则（system prompt）

```
1. 不要编造过去的具体对话或细节
2. 问起以前的 → 单条记忆：还原场景；多条记忆：列选项反问；无记忆：引导分享
3. 提到的细节必须来自真实记忆记录，绝对不能编造
```

### 6.2 上下文规则（每次注入）

```
规则 1-4: 通用回答约束（基于事实、不说"看到"了什么、还原最强维度等）
规则 5:  多条记忆时不猜，列选项反问
规则 6:  线索不足时主动追问
规则 7-8: 资料/文章的处理方式
规则 9:  text 是用户原话，不要概括
规则 10: 情感浓度高时放慢语感
规则 11: 用户提到地点时，只从那地点的记忆中选
```

### 6.3 为什么需要这些规则？

没有规则时，AI 会：
- 从 5 条记忆中随机选一条回答（通常是情感强度最高的那条）
- 编造不在记忆中的细节来"丰富"回答
- 把用户 A 的经历和用户 B 的记忆混淆
- 用"你提到过……"把用户的故事概括化，而不是直接引用

规则的本质是：**让 AI 成为记忆的忠实呈现者，而不是记忆的创作者**。

## 七、线索重建（SceneReconstruction）

### 7.1 设计思路

线索重建是搜索的**前端对称体**。用户在前端输入任意关键词，前端自己算一次相似度匹配（不依赖后端搜索），用来做可视化回忆。

### 7.2 前端匹配算法

```
用户输入 "咖啡厅 淡蓝色衬衫 阳光"

1. nineDEncoder.detectDimensionBoosts() → 检测哪些维度被触发
2. 提取 bigram（双字词）：咖啡、厅淡、蓝色、衬衫、阳光
3. 对每条记忆计算：
   - Jaccard 相似度（bigram vs keywords）权重 0.4
   - 9D 维度匹配（场景/人物/情感/物件等）权重 0.6
4. 阈值 > 0.05 → 显示结果
```

### 7.3 前后端搜索差异

| | 后端 searchDimension() | 前端 SceneReconstruction |
|:---|:---|:---|
| 匹配维度 | 9D 全方位 + 文本 | 主要靠 keyword bigram + 部分9D |
| 评分因子 | 18 个加权因子 | 2 个因子（textSim + dimMatch） |
| 用途 | AI 上下文获取 | 用户可视化探索 |
| 阈值 | 0.15 | 0.05 |

## 十、FAQ

**Q: 为什么不用向量数据库（如 Pinecone/Chroma）？**
A: 当前记忆量 < 300 条，JSON 文件 + 规则评分足够快且简单。等记忆超过 1000 条再考虑向量索引。

**Q: 为什么用 DeepSeek 而不是 Claude？**
A: 初始选择。架构上 DeepSeek 和 Claude 都可以，只需要改 chat.js 中的 API 调用。

**Q: 为什么 9D 拆词元而不是直接用原文？**
A: 9D 拆分的目的是**多角度检索**。原文只能做关键词匹配，而 9D 拆分后可以从场景、人物、情感、物件等独立维度匹配——用户说"黄昏"就能找到场景维度含黄昏的记忆，即使用户没说"黄昏"二字的具体上下文。

**Q: 记忆中文乱码是怎么回事？**
A: PowerShell 5.1 的 `ConvertTo-Json` / `Invoke-WebRequest` 默认 UTF-16 LE 编码，而 API 和 JSON 文件用 UTF-8。用测试脚本时中文变 `????`。生产环境（浏览器 + 前端）没有此问题。
